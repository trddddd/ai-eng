# HW-1: Отчёт по фиче #2 — Content Bootstrap for Study MVP

## 1. Время на формирование комплекта документов (Brief + Spec + Plan)

**Brief — ~15 минут личного времени.**
Написан своими словами, лёгкая «причёска» через AI. Самый быстрый этап.

**Spec — наибольшие затраты времени.**
Инструменты авторевью задавали значительное количество нерелевантных вопросов и систематически пытались раздуть скоуп сверх необходимого. Каждую итерацию приходилось сужать контекст вручную. Несмотря на то что инструмент полезен для выявления пробелов, соотношение сигнал/шум оказалось низким.

**Plan — ~одна итерация + ручная правка.**
После трудоёмкого этапа со спекой план сошёлся быстро. На ревью плана внимание уделялось минимально — в основном обсуждались места без конкретики и выбор решений там, где было несколько вариантов.

## 3. Оценка качества результата

**Оценка: 7/10**

**Что получилось хорошо:**
- Тесты написаны качественно: правильная структура, edge cases, идемпотентность.
- Архитектура слоёв соблюдена: операции, модели, rake tasks — каждый в своём месте.
- Rubocop чистый с нуля после финального прохода.

**Что получилось плохо:**
- **Acceptance Criteria не были прогнаны на чистой БД.** При ручной проверке обнаружилось, что базы пустые — агент не выполнил `db:seed` как часть верификации. Корневая причина: в спеке это было прописано явно, но в плане осталось размытым, и агент не воспринял это как обязательный шаг верификации.
- Часть очевидных вещей потребовала явного указания, хотя была задокументирована в Acceptance Criteria.

---

## 4. Что менять в промптах

**Spec-ревью инструменты:**
Снизить агрессивность расширения скоупа. Агенты-ревьюеры должны задавать вопросы только в рамках явно заявленного скоупа, а не предлагать добавить то, чего нет в брифе. Возможное решение — передавать бриф как hard constraint при запуске ревью.

**Plan:**
Шаг верификации Acceptance Criteria должен быть атомарным и неразмытым. Конкретная формулировка: «запустить `bin/setup --skip-server` на чистой БД, проверить counts» — не «провести ручную верификацию». Это теперь зафиксировано в `AGENTS.md`.

**AGENTS.md / окружение:**
Агент не смотрит файлы в корне проекта автоматически. Нюансы окружения (Docker Compose для БД, `.env`, порты) нужно прописывать явно в `AGENTS.md`, не рассчитывая что агент сам их обнаружит. Добавленные правила в этой сессии закрывают большую часть этого пробела.

**Общий вывод:**
Пайплайн Brief → Spec → Plan → Impl работает, но самым дорогим этапом неожиданно оказалась спецификация, а не реализация. При следующем заходе стоит либо ограничивать глубину spec-ревью, либо настраивать агентов так, чтобы они работали только с тем, что явно заявлено в брифе.
