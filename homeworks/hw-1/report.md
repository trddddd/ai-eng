### 3. Оценка качества: 8/10

Реализация в целом хорошая, но обнаружился баг при ручной проверке: `import_quizword.rb:232` использует `String#include?` — substring match без учёта границ слов. Из-за этого "ago" матчится внутри "agony" и подобных слов.

## Фича 007 — Сессия повторений со спейсед репетишн (FSRS)

### 1. Время на документы (Brief + Spec + Plan)

~3 часа суммарно, растянутых на 2 дня из-за лимитов токенов. Причины:
- Новый подход с vertical slicing потребовал обширных обсуждений
- Spec-ревью промпты были нагруженными и съедали много контекста
- Частые автокомпакты из-за исчерпания лимитов

### 2. Время на имплементацию

| | Время |
|---|---|
| Личное время | ~1 ч |
| Время агента | ~30 мин |

Первый подход к vertical slicing прошёл успешно, но съел весь контекст — автокомпакт привёл к деградации качества.

### 3. Оценка качества: 6.5/10

Основной backend (модели, операции, FSRS-пайплайн) реализован корректно. Проблемы сконцентрировались в UI-слое — 5 расхождений со спекой, потребовавших ручных фиксов после имплементации:

1. **CORS аудио** — Tatoeba не отдаёт CORS-заголовки, пришлось писать серверный прокси (не предусмотрен спекой)
2. **Вложенный turbo_frame** — дублирование frame в show + partial ломало Turbo matching
3. **Ghost text overlay** — `position: absolute` не работал, заменён на `input.placeholder`
4. **Audio + submit race condition** — fire-and-forget audio + повторный триггер submit, потребовался state machine с guards
5. **button_to внутри формы** — невалидный вложенный `<form>`, заменён на `link_to`

Подробности: [memory-bank/features/007/postimplementation.md](../memory-bank/features/007/postimplementation.md)

### 4. Что менять в промптах

**Acceptance Criteria:** Не хватило конкретных AC чтобы фича получилась с первого раза. Без feedback loop агент не смог продумать и отловить все edge cases — они вылезли только при ручной проверке.

**Контекст и автокомпакт:** Первый подход к vertical slicing был успешным, но исчерпание токенов привело к автокомпакту, после которого качество деградировало. Для крупных фич нужна стратегия управления контекстом: либо разбивать на отдельные сессии по слайсам, либо закладывать checkpoint'ы в план.

**Рекомендации из postimplementation:**
- При интеграции с внешними ресурсами — проверять CORS до спеки
- Явно указывать ownership turbo_frame (partial vs parent)
- Для async UI flows — описывать как state machine, не как sequence of calls
- Предпочитать нативные HTML-механизмы кастомным overlay

## Выводы

### Что работает

- **Пайплайн Brief → Spec → Plan → Impl** стабильно работает для фич малого и среднего размера (004–006: качество 8–9/10).
- **Декомпозиция** крупных фич на подготовительные (006) и основные (007) — правильная стратегия. 006 прошла гладко, 007 была бы ещё тяжелее без неё.
- **Время на документы** окупается: фичи 005–006 реализовались за 30 минут с минимальным участием.

### Что нужно улучшить

1. **AC должны содержать конкретные edge-case примеры** — без них агент не отлавливает граничные случаи (004: substring match; 007: CORS, race conditions).
2. **Управление контекстом для крупных фич** — автокомпакт убивает качество. Стратегия: один слайс = одна сессия, с checkpoint'ами между сессиями.
3. **UI/browser-специфичные кейсы** плохо покрываются спекой — CORS, вложенные формы, async playback. Нужен отдельный чеклист для frontend AC.
4. **Spec-ревью для крупных фич** съедает слишком много токенов — нужно либо ограничивать глубину, либо ревьюить послайсово.
