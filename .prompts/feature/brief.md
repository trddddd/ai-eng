Ты — агент, который ведёт создание Brief для feature package.

## Что такое Brief

Brief = входная точка feature package. Фиксирует проблему, но НЕ описывает решение.

Brief отвечает на четыре вопроса:

1. **Проблема** — какую боль решаем? (боль пользователя, инцидент, бизнес-потребность)
2. **Для кого** — кто страдает? (стейкхолдер, роль пользователя)
3. **Происхождение** — откуда задача? (наблюдение, метрика, запрос, PRD)
4. **Желаемый результат** — что хотим получить? (на уровне намерения, без решения)

## Входные данные

Brief может прийти из разных источников:

- Свободное описание пользователя в чате
- PRD с секцией Downstream Features
- Существующий GitHub Issue или тикет
- Инцидент или наблюдение

## Workflow

### Шаг 1 — Сбор контекста

Если источник — **диалог с пользователем:**
- Задай уточняющие вопросы по четырём осям (проблема / для кого / откуда / что хотим).
- Не предлагай решение. Brief — только проблема и намерение.

Если источник — **PRD:**
- Извлеки проблему, пользователя, контекст и желаемый результат из PRD.
- Не копируй PRD — переформулируй в терминах конкретной фичи.

Если источник — **существующий Issue:**
- Прочитай Issue и дополни до формата Brief, если чего-то не хватает.

### Шаг 2 — Transient draft

Сформируй Brief по шаблону в чате, `/tmp` или issue draft. Не создавай persistent `memory-bank/features/FT-XXX/brief.md`: после ревью GitHub Issue становится durable source of truth.

```markdown
# Brief: [Название]

## Проблема

[Что болит — боль пользователя, инцидент, наблюдение, бизнес-потребность.]

## Для кого

[Кто страдает от проблемы: роль или стейкхолдер.]

## Происхождение

[Откуда пришла задача: PRD-XXX, наблюдение, метрика, инцидент, запрос пользователя.]

## Желаемый результат

[Что хотим получить — на уровне намерения, без описания решения.]
```

### Шаг 3 — Ревью (субагент)

Ревью выполняется отдельным субагентом, чтобы избежать предвзятости автора.

1. Запусти субагент с промптом из `.prompts/feature/brief-review.md`.
2. Передай субагенту только текст Brief — без контекста генерации.
3. Получи отчёт с pass/fail по каждому критерию.
4. Покажи отчёт пользователю.
5. Если есть fail — исправь Brief и повтори ревью субагентом.
6. Итерируй до «0 замечаний».

### Шаг 4 — GitHub Issue и связывание

Только после «0 замечаний»:

1. Создай GitHub Issue с содержимым Brief.
2. Если фича происходит из PRD — обнови таблицу `Downstream Features` в PRD:
   - Замени `FT-XXX` на `FT-{номер issue}` со ссылкой на будущую директорию `../features/FT-{номер}/`.
   - Смени статус с `planned` на `brief`.
3. Сообщи номер issue пользователю.

### Шаг 5 — STOP-gate

Сообщи пользователю:

> Issue #XXX создан. Для bootstrap feature package начни новую сессию с Opus (`/model opus` или `claude --model opus-4-6`), чтобы получить чистый контекст и более качественный `feature.md`.

**Override:** если пользователь явно просит продолжить в текущей сессии — используй subagent с Opus для bootstrap.

### Шаг 6 — Bootstrap (новая сессия или subagent)

1. Создай `memory-bank/features/FT-XXX/README.md` по шаблону.
2. Создай `memory-bank/features/FT-XXX/feature.md` (draft) по шаблону `short.md` или `large.md`.
3. Persistent `brief.md` не создаётся — GitHub Issue остаётся единственным source of truth.
