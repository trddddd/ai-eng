# Brief: Дизайн-система проекта

**Issue:** trddddd/ai-eng#16
**State:** SHIPPED

## Проблема

В проекте нет единого описания визуального языка: цветовой палитры, типографики, отступов, состояний компонентов. UI собирается ad hoc — каждый новый экран использует произвольные стили напрямую, без привязки к общим визуальным токенам или правилам. Это приводит к:

- **Визуальной рассогласованности** — разные экраны выглядят как разные продукты (разные оттенки серого для фонов, разные радиусы скруглений, разные отступы).
- **Замедлению разработки** — каждый раз при создании нового экрана или компонента приходится заново принимать решения о цветах, шрифтах, размерах, вместо того чтобы обратиться к готовому набору правил.
- **Невозможности делегирования** — AI-агенты и другие разработчики не имеют ориентира для принятия визуальных решений; результат непредсказуем.

## Для кого

- **Разработчик (включая AI-агентов)** — при создании новых экранов и компонентов нужен единый справочник визуальных решений, чтобы результат был консистентным без ручного ревью каждого пикселя.
- **Пользователь (learner)** — единообразный интерфейс снижает когнитивную нагрузку и формирует доверие к продукту.

## Откуда взялась задача

Проект прошёл стадию прототипирования (карточки, сессии повторений, онбординг). Количество экранов растёт; визуальная согласованность между ними падает. CSS-фреймворк используется, но без кастомизации темы и без документированных правил — каждый экран собирается с нуля.

## Что мы хотим получить

При создании нового экрана или компонента разработчик (или AI-агент) обращается к единому справочнику визуальных решений проекта и получает однозначные ответы на вопросы: какой цвет использовать, какой размер шрифта, какие отступы, как выглядит кнопка в каждом состоянии.

После принятия справочника — редизайн экрана сессии повторений (карточка cloze) как первого экрана, приведённого к новой системе. Этот экран служит валидацией: если по справочнику можно пересобрать основной пользовательский сценарий без дополнительных визуальных решений, справочник работает.

Критерий готовности: по справочнику можно однозначно определить визуальное оформление любого UI-примитива (кнопка, инпут, карточка, алерт) в каждом из его состояний (default, hover, focus, disabled, error) — без обращения к другим экранам за образцом.

## Вне скоупа

- Реализация компонентной библиотеки — отдельная задача.
- Редизайн остальных экранов под новую систему — отдельная задача после валидации на экране сессии.
- Branding (логотип, айдентика, тон коммуникации) — не входит.
- Создание отдельного конфигурационного файла CSS-фреймворка (`tailwind.config.js`) — токены определяются через `@theme` в CSS.
- RTL-языки — не поддерживаются на данном этапе.
