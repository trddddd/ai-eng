# Brief Template

Brief — входная точка feature package. Фиксирует проблему, но не описывает решение.

## Workflow создания брифа

1. Пользователь описывает идею свободно в чате.
2. Агент задаёт уточняющие вопросы по четырём осям (проблема / для кого / откуда / что хотим).
3. После согласования — агент создаёт черновик `brief.md` (без issue-ссылки, номер ещё неизвестен).
4. **Агент прогоняет бриф через ревью (см. ниже) и фиксирует замечания в чате — построчно по каждому из 5 критериев, с явным pass/fail.** До вызова `gh issue create` в чате должно быть видно «0 замечаний, Brief готов к работе».
5. Если замечания есть — агент исправляет `brief.md` и повторяет ревью.
6. Только после «0 замечаний» — агент создаёт GitHub Issue.
7. ⛔ **STOP-gate.** Агент сообщает пользователю номер issue и предлагает начать новую сессию с Opus для bootstrapping:
   > Issue #XXX создан. Для bootstrapping feature package начни новую сессию командой `/model opus` (или `claude --model opus-4-6`), чтобы получить чистый контекст и более качественный `feature.md`.
8. **Override:** Если пользователь явно запросил продолжение в текущей сессии (например, «делай просто сабагентом»), STOP-gate снимается. Агент использует subagent с Opus для bootstrapping вместо новой сессии. При этом brief.md всё равно удаляется — GitHub Issue остаётся единственным source of truth.
9. **[новая сессия Opus или subagent]** Агент удаляет `brief.md`, оставляя только GitHub Issue.
10. **[новая сессия Opus или subagent]** Агент bootstraps feature package: `README.md` + `feature.md` (draft).

## Ревью брифа

Перед bootstrap feature package агент проверяет `brief.md` по пяти критериям:

1. Проблема конкретна и измерима (не «улучшить», а конкретная метрика или боль).
2. Назван стейкхолдер или пользователь, для которого решаем.
3. Понятен контекст: откуда задача, почему важна сейчас.
4. Brief НЕ содержит решения — только проблему и желаемый результат.
5. Нет двусмысленных формулировок («быстро», «удобно», «при необходимости», «осмысленный» и т.п.).

Для каждого замечания агент указывает:
- Что именно не так (цитата из документа).
- Почему это проблема.
- Как исправить (конкретное предложение).

Если замечаний нет — агент пишет «0 замечаний, Brief готов к работе» и переходит к bootstrap.

**Exception для тривиальных фиксов:** если change surface ≤ 3 строк и fix очевиден из описания проблемы (например, «убрать кавычки»), критерий 4 ослабляется — допустимо упоминать очевидный уровень решения. Агент помечает это как «тривиальный fix, solution hints допустимы».

## Уточняющие вопросы агента

Агент должен прояснить:

- **Проблема:** что конкретно не работает или причиняет боль? Есть ли пример из реальности?
- **Для кого:** кто именно страдает — изучающий язык, администратор, система?
- **Происхождение:** это наблюдение, запрос пользователя, метрика, технический долг?
- **Результат:** если проблема решена — что изменится? Как это будет выглядеть?

Агент не должен предлагать решение на этом этапе. Brief — только проблема и намерение.

## Instantiated Body

```markdown
# Brief: [Название]

> GitHub Issue: [owner/repo#XXX](https://github.com/owner/repo/issues/XXX)

## Проблема

[Что болит — боль пользователя, инцидент, наблюдение, бизнес-потребность.]

## Для кого

[Кто страдает от проблемы: роль или стейкхолдер.]

## Происхождение

[Откуда пришла задача: наблюдение в чате, метрика, инцидент, запрос пользователя.]

## Желаемый результат

[Что хотим получить — на уровне намерения, без описания решения.]
```
